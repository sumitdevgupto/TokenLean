# Client Onboarding — TokenLean (Operator Runbook)

**Audience:** the platform operator onboarding a new client onto the managed (Enterprise) SaaS.
Every step below is a real prod action with a **local mirror** command so it can be rehearsed on the
local commercial stack before running it against GCP. Follow it top to bottom and a client goes from
"nothing" to "sending optimised traffic and seeing their savings" — with full operator oversight.

> The managed SaaS runs **strict BYOK**: each client brings their own LLM provider keys. Under
> enforcement, a client's requests return **HTTP 402** until they add at least one provider key. This
> is intentional — the platform never bills its own provider account for client traffic.

---

## 0. Prerequisites & environment

| Thing | GCP (prod) | Local mirror |
|---|---|---|
| Portal URL | `https://portal-svc-<hash>-<region>.run.app/portal/` | `http://localhost:8090/portal/` |
| Proxy / LLM endpoint | `https://token-proxy-<hash>-<region>.run.app` | `http://localhost:4000` |
| BYOK enforcement | `BYOK_ENFORCE=true` (set by `deploy-commercial-gcp.sh`) | set it on for a faithful run (Step 0.1) |
| Master key | Secret `tenant-key-encryption-key` | `.env` `TENANT_KEY_ENCRYPTION_KEY` (auto-generated by the local deploy) |
| Admin key | an admin proxy key held by the operator | mint one (Step 0.2) |

> **How this runbook handles the two environments.** Only the **setup** steps (0.0–0.4: deploy,
> BYOK toggle, admin-key mint, URL discovery, email/log access) genuinely differ between GCP and
> local — each of those shows an explicit **GCP** and **Local** command. **Steps 1–7 are identical
> in both environments**: they only use the `$PORTAL` / `$PROXY` / `$ADMIN_KEY` shell variables set
> in Step 0.3, so the same command works verbatim against Cloud Run or Docker.

### 0.0 Deploy

**GCP:**
```bash
bash scripts/commercial/deploy-commercial-gcp.sh          # first run provisions everything
# (sets BYOK_ENFORCE=true, binds the tenant-key-encryption-key secret, patches proxy_base_url)
```
**Local:**
```bash
cd d:/token-optimisation
bash scripts/commercial/deploy-commercial-local.sh --recreate
```

> **Real verification emails are a property of the deploy — no separate step.** When
> `SENDGRID_API_KEY=SG.…` is in `.env`, the compose overlay forwards it into the proxy
> (`SENDGRID_API_KEY: ${SENDGRID_API_KEY:-}`) and `up -d` recreates the proxy whenever that
> value changes, so signup sends live email out of the box. The deploy prints which sender is
> active: `Email: SendGrid (real delivery)` or `Email: Fake sender (no SENDGRID_API_KEY…)` —
> with the Fake sender the verification link is only written to the logs (read it via §0.4),
> never emailed. **GCP** binds the key as a Secret Manager secret in `deploy-commercial-gcp.sh`.
> If you change `.env` or the compose overlay while the stack is already up, just re-run the
> deploy — `up -d` reconciles the one changed service.
>
> **No email despite a 202?** The signup `202` only means the verification token was created — not
> that SendGrid accepted the message. Grep the proxy logs for the send result:
> `docker logs token-opt-proxy 2>&1 | grep -i "SendGrid send"`. Common failures: `401 Maximum credits
> exceeded` (account out of credits — add credits or a funded key), `403` (from-address is not a
> verified SendGrid Sender Identity). A credit/auth rejection never reaches the SendGrid Activity Feed.

### 0.1 Enable BYOK enforcement (mirror prod)

**GCP:** already on — `deploy-commercial-gcp.sh` sets `BYOK_ENFORCE=true` on the token-proxy service.
To toggle it on an existing service without a full redeploy:
```bash
gcloud run services update token-proxy --region "$REGION" --project "$PROJECT_ID" \
  --update-env-vars BYOK_ENFORCE=true
```
**Local:** pass `--byok-enforce` — it persists the setting to `.env` (not a transient shell export)
so it survives across shells and every later deploy, exactly like the Cloud Run env var above:
```bash
cd d:/token-optimisation
bash scripts/commercial/deploy-commercial-local.sh --byok-enforce
```
Confirm — expect `BYOK_ENFORCE=true`, the resolver line, and `health: 200`:
```bash
# Local:
grep '^BYOK_ENFORCE' .env && docker logs token-opt-proxy 2>&1 | grep -i "BYOK" | tail -2 \
  && curl -s -o /dev/null -w 'health: %{http_code}\n' http://localhost:4000/health
# GCP:
gcloud run services describe token-proxy --region "$REGION" --format='value(spec.template.spec.containers[0].env)' | tr ',' '\n' | grep BYOK
gcloud run services logs read token-proxy --region "$REGION" --limit 200 | grep -i BYOK | tail -2
```
> The `BYOK: tenant-aware provider key resolver installed` line appears **once per worker**
> (twice on the default local 2-worker stack) — that's expected, not a duplicate.
>
> `--byok-enforce false` (local) / `--update-env-vars BYOK_ENFORCE=false` (GCP) flips it back off.

## 1. Provision the tenant

Two paths. **Path A (self-serve)** is the default — the client does it themselves in the browser.
**Path B (operator-assisted)** is for enterprise clients the operator provisions directly.

### Path A — self-serve signup (default)

The client does this in the browser at the Portal URL; the equivalent API calls (identical for GCP
and local — they use the Step 0.3 vars) are shown so the operator can verify each stage and so the
automation can mirror them.

**1A.1 — Request a verification email** (same call in both environments)

```bash
curl -s -X POST "$PORTAL/portal/auth/signup" \
  -H 'Content-Type: application/json' \
  -d '{"email":"owner@acme.com"}'
# → {"status":"verification_sent"}   (uniform response; never reveals if the email already exists)
```

Retrieving the verification token differs only in *where the email goes*:
- **GCP:** a real email is sent via SendGrid — the client clicks the link in their inbox (the
  `?token=…` query param is the value below). Operators without inbox access can read it from logs:
  ```bash
  gcloud run services logs read token-proxy --region "$REGION" --limit 200 | grep -oE 'token=[A-Za-z0-9_-]+' | tail -1
  ```
- **Local (no SendGrid):** the Fake sender **logs** the link instead of emailing it:
  ```bash
  docker logs token-opt-proxy 2>&1 | grep -oE 'token=[A-Za-z0-9_-]+' | tail -1
  # → token=<VERIFY_TOKEN>
  ```

**1A.2 — Validate the link**

```bash
VERIFY_TOKEN=...   # from the email / logs
curl -s "$PORTAL/portal/auth/verify?token=$VERIFY_TOKEN"
# → {"email":"owner@acme.com"}
```

**1A.3 — Complete signup (set password + company + code)**

The client picks a **company code** — 4 letters, auto-suggested from the company name (Acme → `ACME`),
editable, unique platform-wide. The first stack becomes `{CODE4}-PRD-01`.

```bash
curl -s -X POST "$PORTAL/portal/auth/complete-signup" \
  -H 'Content-Type: application/json' \
  -d '{"token":"'"$VERIFY_TOKEN"'","password":"correct-horse-battery","full_name":"Ada Owner","company":"Acme Inc","company_code":"ACME"}'
```

Returns (shown **once**):

```json
{
  "tenant_id": "ACME-PRD-01",
  "company": "Acme Inc",
  "company_code": "ACME",
  "email": "owner@acme.com",
  "api_key": "tok-…",
  "proxy_base_url": "http://localhost:4000",
  "snippet": "client = OpenAI(base_url=\"http://localhost:4000/v1\", api_key=\"tok-…\")",
  "warning": "Store this tenant key now — it is shown once and cannot be recovered."
}
```

The portal header now shows **Acme Inc · ACME-PRD-01**. Save the key:

```bash
TENANT_KEY=tok-...    # from api_key above
```

### Path B — operator-assisted (enterprise)

> Same commands for GCP & local — they use the `$PROXY` / `$ADMIN_KEY` vars from Step 0.3.

Operator creates the tenant + first portal user with the admin key.

```bash
# Create the tenant + issue its first proxy key (id must match {CODE4}-{PRD|STG}-NN)
curl -s -X POST "$PROXY/api/v1/admin/tenants" \
  -H "Authorization: Bearer $ADMIN_KEY" -H 'Content-Type: application/json' \
  -d '{"tenant_id":"ACME-PRD-01","tier":"enterprise","company_name":"Acme Inc"}'
# → {tenant_id, tier, api_key (once), ...}

# Create the client's portal login (returns a temporary password)
curl -s -X POST "$PROXY/api/v1/admin/tenants/ACME-PRD-01/users" \
  -H "Authorization: Bearer $ADMIN_KEY" -H 'Content-Type: application/json' \
  -d '{"email":"owner@acme.com"}'
# → {email, temporary_password}   (share out-of-band; client changes it on first login)

# Need a second stack later? Ask for the next id:
curl -s "$PROXY/api/v1/admin/companies/ACME/next-stack?env=STG" -H "Authorization: Bearer $ADMIN_KEY"
# → {"next_stack_id":"ACME-STG-01"}
```

## 2. First request — see enforcement in action

> Identical for GCP & local (uses the Step 0.3 vars).

Under BYOK enforcement, before the client adds a provider key their traffic is **refused with 402**:

```bash
curl -s -o /dev/null -w '%{http_code}\n' -X POST "$PROXY/v1/chat/completions" \
  -H "Authorization: Bearer $TENANT_KEY" -H 'Content-Type: application/json' \
  -d '{"model":"gpt-4o-mini","messages":[{"role":"user","content":"hi"}],"max_tokens":5}'
# → 402   (body: {"error":{"code":"provider_key_missing", ...}})
```

This is the expected, correct behaviour — it proves the platform will not bill its own account.

## 3. Client adds their provider key (BYOK)

> Identical for GCP & local (uses the Step 0.3 vars).

In the portal: **Settings → Models & Keys → add key**. API equivalent (values are write-only; only
the last 4 are ever returned):

```bash
curl -s -X PUT "$PROXY/portal/provider-keys/openai" \
  -H "Authorization: Bearer $TENANT_KEY" -H 'Content-Type: application/json' \
  -d '{"api_key":"sk-<client-real-openai-key>"}'
# → {"provider":"openai","last4":"…","replaced":false,"note":"Stored encrypted. Takes effect within ~60s."}

# Confirm (never shows the value):
curl -s "$PROXY/portal/provider-keys" -H "Authorization: Bearer $TENANT_KEY"
```

> Keys are Fernet-encrypted at rest (`tenant_provider_keys`). Propagation is ~60 s (key-cache TTL).

## 4. Client sets model preferences (optional)

> Identical for GCP & local (uses the Step 0.3 vars).

Default model + the three cascade-tier head models. A field left empty reverts to the platform
default. In the portal: **Settings → Models & Keys → model preferences**. API:

```bash
curl -s -X POST "$PROXY/portal/models/config" \
  -H "Authorization: Bearer $TENANT_KEY" -H 'Content-Type: application/json' \
  -d '{"default_model":"gpt-4o-mini","tiers":{"simple":"gpt-4o-mini"}}'

# Effective view (merged base + overrides + key status + BYOK state):
curl -s "$PROXY/portal/models" -H "Authorization: Bearer $TENANT_KEY"
```

## 5. Integrate — the one-line change

The client points their existing OpenAI-compatible SDK at the proxy. **Only the base URL + key
change**; all G0–G28 optimisations apply transparently.

```python
from openai import OpenAI
client = OpenAI(
    base_url="http://localhost:4000/v1",   # PROXY endpoint (from signup / Models & Keys)
    api_key="tok-…",                        # tenant proxy key
)
resp = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "Summarise our refund policy."}],
    extra_headers={"X-User-ID": "ada@acme.com"},   # per-user attribution (must match your domain)
)
```

Raw first call:

```bash
curl -s -X POST "$PROXY/v1/chat/completions" \
  -H "Authorization: Bearer $TENANT_KEY" -H 'Content-Type: application/json' \
  -H 'X-User-ID: ada@acme.com' \
  -d '{"model":"gpt-4o-mini","messages":[{"role":"user","content":"Say hello in 3 words"}],"max_tokens":16}'
# → 200 with the completion + a "_token_opt" savings block
```

> **X-User-ID** must match the tenant's own email domain (`*@acme.com`, stamped at signup) or the
> configured allowlist, otherwise attribution is dropped. This prevents one client booking usage
> under another's user id.

## 6. Verify the onboarding

> Identical for GCP & local (uses the Step 0.3 vars).

### Client-visible (tenant key)

> The `usage_events` row is written by a fire-and-forget task **after** the 200 returns, so give it
> a second (or poll) before expecting the just-made call in the ledger.

```bash
# Per-call ledger: tokens that hit the proxy → sent to the LLM → provider-billed, + cost
curl -s "$PROXY/portal/usage/calls?limit=5" -H "Authorization: Bearer $TENANT_KEY"

# Rollup by user / day / model / provider
curl -s "$PROXY/portal/usage/rollup?group_by=user,day" -H "Authorization: Bearer $TENANT_KEY"

# Billing (request count) + disclosed savings + quota consumption
curl -s "$PROXY/portal/usage" -H "Authorization: Bearer $TENANT_KEY"
```

### Operator oversight (admin key)

```bash
# Effective config + provider-key status + quota for one tenant
curl -s "$PROXY/api/v1/admin/tenants/ACME-PRD-01/config" -H "Authorization: Bearer $ADMIN_KEY"

# Config-change audit trail (provider_key.set, config.model_prefs_updated, …)
curl -s "$PROXY/api/v1/admin/tenants/ACME-PRD-01/audit?limit=20" -H "Authorization: Bearer $ADMIN_KEY"

# Cross-tenant usage: which tenant burns which provider/model, via which user
curl -s "$PROXY/api/v1/admin/usage/rollup?group_by=tenant,provider" -H "Authorization: Bearer $ADMIN_KEY"
```

**Onboarding is complete when:** the client's chat returns 200, the call appears in
`/portal/usage/calls` with the token triplet + cost, and the admin config view shows the tenant's
provider key + effective model.

## 7. Ongoing lifecycle (reference)

| Action | Who | Call |
|---|---|---|
| Rotate proxy key (compromise / policy) | client | `POST /portal/proxy-key/rotate` (new key once; all old revoked) |
| Suspend / unsuspend | operator | `POST /api/v1/admin/tenants/{tid}/suspend` · `/unsuspend` |
| Add another stack (`ACME-STG-01`) | operator | `POST /api/v1/admin/tenants` with the next id |
| Monthly quota gate | operator | `rate_limit.quota.enabled` + `billing.rate_card.<tier>.included_requests` → 429 `quota_exceeded` |
| Offboard (revoke + erase + free code) | operator | `POST /api/v1/admin/tenants/{tid}/offboard` |
| GDPR export / erase | operator | `GET`/`DELETE /api/v1/tenants/{tid}/data` |

## Appendix A — reading the savings block

Every response includes a `_token_opt` object:

```json
{
  "choices": [...],
  "_token_opt": {
    "baseline_tokens": 450,
    "final_tokens_sent": 220,
    "total_pct_saving": 51.1,
    "cache_hit": false,
    "routed_model": "gpt-4o-mini",
    "cost_baseline_usd": 0.002250,
    "cost_actual_usd": 0.000033,
    "step_savings": { "G01": {"abs_saving": 85}, "G10": {"abs_saving": 145} }
  }
}
```

> **Fair-disclosure:** `cost_*_usd` are **config-priced estimates** (token counts × the static
> `pricing:` table) — directional, not invoice-grade. Billing is **request-count**, never tokens.
> Token counts are exact.

## Appendix C — document ingestion for RAG (G03)

The proxy has no direct upload endpoint. Documents are ingested when they land in the configured GCS

ingest bucket + write access. Full flow: [request-flow-diagram.md](request-flow-diagram.md#document-ingestion-pipeline-g03).

## Automation

The whole runbook is wrapped in one operator script — `scripts/commercial/onboard-client.sh`
(commercial, gitignored). It runs **either** the whole flow in one shot **or** one step at a time,
with an assertion after each step.

**One step at a time** (the operator drive — one named flag per step, matching the §N above). Each
runs a single step and persists its outputs (admin key, verify token, tenant key, …) to a state file
(`scripts/commercial/.onboard-state.env`), so the next continues where it left off. On completion each
step prints the flag for the next one.

```bash
# ── operator setup (once, not per client) ──
bash scripts/commercial/onboard-client.sh --byok           # §0.1 BYOK posture
bash scripts/commercial/onboard-client.sh --admin-key      # §0.2 mint the operator admin key
# ── client onboarding → PORTAL ACCESS (the core sequence) ──
bash scripts/commercial/onboard-client.sh --signup --file scripts/commercial/clients/acme.env
                                                           # §1.1 client requests access → verification email
bash scripts/commercial/onboard-client.sh --verify-email   # §1.2 client clicks the emailed link
bash scripts/commercial/onboard-client.sh --complete       # §1.3 sets password → tenant key + endpoint
# ── client self-service (the CLIENT does these IN THE PORTAL after onboarding) ──
bash scripts/commercial/onboard-client.sh --provider-key   # §3   client adds their BYOK key
bash scripts/commercial/onboard-client.sh --model-prefs    # §4   client sets model preferences
# ── usage + verify ──
bash scripts/commercial/onboard-client.sh --test-402       # §2   request refused 402 before a key exists
bash scripts/commercial/onboard-client.sh --first-call     # §5   first optimised request
bash scripts/commercial/onboard-client.sh --oversight      # §6   verify ledger + admin oversight
bash scripts/commercial/onboard-client.sh --offboard       # §7   (optional) offboard + clear state
```

**Onboarding = portal access only** (`--signup` → `--verify-email` → `--complete`). Provider keys,
model preferences, and user IDs are **not** onboarding inputs — the client configures those in the
portal after they have access (or sends `X-User-ID` per request). The onboarding inputs file
therefore only needs `COMPANY_NAME`, `COMPANY_CODE`, `OWNER_EMAIL`, `PASSWORD`.

(`--step <name|0.2>` is also accepted — numeric §-IDs are aliases.)

**Client inputs come from a file** (recommended for real onboarding) — one spec per client, passed
once on `--key-identity` and remembered for the later steps:

```bash
# scripts/commercial/clients/acme.env  (gitignored; copy from client.env.example)
COMPANY_NAME="Acme Inc"   COMPANY_CODE=ACME   OWNER_EMAIL=owner@acme.com
PASSWORD=…   USER_ID=ada@acme.com   MODEL=gpt-4o-mini
CLIENT_PROVIDER=openai   CLIENT_PROVIDER_KEY=sk-…

bash scripts/commercial/onboard-client.sh --signup --file scripts/commercial/clients/acme.env
```

The provider key stays in the file (never written to the state file). Without `--file`, fields fall
back to env vars, then to auto-generation (a random `{CODE}-PRD-01` throwaway tenant).

**All in one** (rehearsal or bulk onboarding):

```bash
bash scripts/commercial/onboard-client.sh --cleanup    # throwaway tenant, verify all, offboard
OWNER_EMAIL=owner@acme.com COMPANY_NAME="Acme Inc" COMPANY_CODE=ACME \
  CLIENT_PROVIDER_KEY=sk-… bash scripts/commercial/onboard-client.sh   # real onboarding, tenant persists
```

Both modes are environment-driven: set `PROXY` / `PORTAL` to the Cloud Run URLs, pass `ADMIN_KEY` and
`VERIFY_TOKEN` (from the client's real email), and the same script onboards against GCP — see the
script header for the exact `gcloud` invocation. Client inputs (`OWNER_EMAIL`, `COMPANY_CODE`,
`CLIENT_PROVIDER_KEY`, …) are auto-generated for a test run if unset.

**Reset a test signup (re-testing).** Signup returns "you already have an account" for an email that
already has a portal login — it never re-sends a link. Free it first so the address (and its CODE4) can
sign up clean:
```bash
bash scripts/commercial/purge-signup.sh --email owner@acme.com
```
It deletes the login + releases the company code. For a **structured** self-serve tenant (`{CODE4}-PRD-NN`)
it also purges that tenant's provider keys / configs / usage / proxy-keys — a same-code re-signup reuses
the same tenant id, so stale data would otherwise carry into the "fresh" run. A **legacy** tenant's data is
left intact (`--force` purges it too; `--keep-tenant-data` frees only the login).

**Full automated lifecycle test.** `scripts/commercial/developer-onboarding-lifecycle-test.sh` runs the
entire flow end-to-end against a throwaway tenant and asserts each stage (preflight · admin key · onboard ·
402 gate · provider key · model prefs · first call · warm-cache hit · oversight ledger/admin/audit · key
rotation · suspend/unsuspend · offboard teardown), printing a PASS/FAIL summary and cleaning up on exit:
```bash
bash scripts/commercial/developer-onboarding-lifecycle-test.sh          # ADMIN_KEY=… to skip the mint wait; --keep to leave the tenant
```
It uses the self-serve signup→verify→complete path when the **Fake** email sender is active (scrapes the
verify token from the proxy logs); with SendGrid active it falls back to operator-assisted tenant creation
(the emailed token can't be scraped) and notes the skip. Live revoke/suspend checks are bounded polls
(store/DB asserts are authoritative — the file key-store propagates within the 300s cache TTL across workers).
